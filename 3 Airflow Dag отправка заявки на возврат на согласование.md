```примечание
Для асинхронного взаимодействия через Kafka используется контракт события (Event Contract). В отличие от REST, здесь фиксируются не запрос/ответ, а топик, схема сообщения, гарантии доставки и роли систем. 
Документ строится так, чтобы продюсер (Orders) и консьюмер (Adapter) могли работать независимо, опираясь на единый источник правды.
```

# 📌Метаданные события

Версия схемы: `v1.0`
Продюсер: `Orders Microservice`
Консьюмер: `Partner Adapter Microservice`
Триггер: Airflow DAG (cron выражение) (https://devmark.ru/tools/cron)
Описание: Уведомляет адаптер о новых заявках на возврат для дальнейшей отправки партнеру.

Пример заполнения: `DAG → SELECT WHERE status='new' → publish → ACK → update status='processing'`

- **Продюсер (Airflow + Orders-MS):**
краткое описание + дать ссылку на подробное описание
    - DAG выполняет `SELECT * FROM returns WHERE status = 'new' AND processed_at IS NULL`.
    - Для каждой записи формирует Kafka-сообщение, публикует в `orders.returns.new`.
    - При успешном `ACK` обновляет статус возврата на `processing` и фиксирует `processed_at`.
- **Консьюмер (Partner Adapter):**
краткое описание + дать ссылку на подробное описание
    - Подписывается на топик, читает сообщения группой консьюмеров.
    - Валидирует `Value` по JSON Schema. При ошибке → отправка в DLQ.
    - Вызывает API партнера, получает результат, публикует ответный статус в `partners.returns.status`.`
# 📌Гарантии и ошибки
```примечание
Что описывать: Delivery semantics, retry policy, DLQ, порядок сообщений
Зачем нужен: Предотвращает потерю данных и дубли
```
Пример заполнения: `At-least-once`, `3 retries`, `DLQ: orders.returns.failed`, `Idempotency by returnId`

| Аспект              | Правило                                                              |
| ------------------- | -------------------------------------------------------------------- |
| `Delivery`          | At-least-once (повторы возможны)                                     |
| `Идемпотентность`   | Консьюмер проверяет `returnId` в БД. Если уже обработан → игнорирует |
| `Retry Policy`      | 3 попытки с экспоненциальной задержкой (1с → 4с → 16с)               |
| `Dead Letter Queue` | `orders.returns.failed` (сохраняет сырое сообщение + причину)        |
| `Порядок`           | Гарантируется только внутри одного `partition key` (`returnId`)      |
