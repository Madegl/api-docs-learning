Процесс делится на два изолированных этапа: 
- **асинхронная доставка** (Airflow DAG паблишит сообщение в Kafka). Dag пишет сообщение в publisher Kafka, а МС "Адаптер" - читает
- **синхронная интеграция** (Адаптер читает Kafka, маппит поля, вызывает REST партнера, обрабатывает ответ). 

`В системном анализе это описывается как Event-to-REST Contract, где фиксируются схема события, правила трансформации данных, гарантии вызова и матрица обработки ответов.`

# **1. Введение**

## 🎯 Бизнес-цель
Автоматизировать передачу заявок на возврат из внутренней системы заказов во внешнюю систему партнёра для:
- Сокращения времени обработки возврата.
- Исключения ручного ввода данных оператором.
- Гарантии идемпотентности: один возврат = один запрос к партнёру.

## 🔗 Участники процесса
| Система | Роль | Ответственность |
|---------|------|-----------------|
| `Orders Microservice` | Продюсер события | Хранит возвраты, предоставляет данные по возврату |
| `Airflow` | Оркестратор | Периодически выбирает новые возвраты, паблишит в Kafka |
| `Kafka (orders.returns.new)` | Шина событий | Гарантирует доставку и порядок обработки по `returnId` |
| `Partner Adapter MS` | Консьюмер + Интегратор | Читает Kafka, маппит поля, вызывает REST партнёра |
| `Partner API` | Внешний получатель | Принимает запрос, валидирует, возвращает статус |

```
## ✅ Критерии успеха

Процесс считается завершённым успешно, если:

1. Возврат в БД `orders` перешёл в статус `submitted_to_partner`.
2. Партнёр вернул HTTP `2xx` и `partnerReturnId`.
3. В лог записано событие `ReturnSentToPartner` с `correlation_id`.

## 🚫 Границы процесса (Out of Scope)

- Обработка вебхуков/колбэков от партнёра о финальном статусе возврата (отдельный процесс).
- Возврат денег клиенту — инициируется после получения подтверждения от партнёра, но не в рамках этого потока.
- Повторная отправка при `4xx`-ошибках валидации (считается бизнес-отказом, требует ручного разбора).
```


# **2. Маппинг Kafka → REST Партнера**

| Поле Kafka (Адаптер получает) | Поле REST (Адаптер отправляет) | Правило трансформации                                                           |
| ----------------------------- | ------------------------------ | ------------------------------------------------------------------------------- |
| `returnId`                    | `externalRequestId`            | Прямое копирование. Используется партнером для корреляции                       |
| `orderId`                     | `originalOrderRef`             | Добавляется префикс: `ORD-` + значение                                          |
| `items[].sku`                 | `items[].productCode`          | Lookup в маппинг-таблице: наш SKU → внешний код партнера                        |
| `totalRefund`                 | `refundAmount`                 | Конвертация в копейки/центы (`* 100`), если партнер требует integer             |
| `createdAt`                   | `requestTimestamp`             | Формат `YYYY-MM-DDTHH:mm:ss.SSSZ` → `epoch_ms` или строка по стандарту партнера |

# **3. Логика обработки ответа партнера**

|Ответ партнера|Действие Адаптера|Статус возврата в БД|
|---|---|---|
|`200 OK` / `201 Created`|Парсинг `partnerReturnId`, запись в лог/БД|`submitted_to_partner`|
|`4xx (ValidationError, Duplicate)`|Запись ошибки, повтор не требуется (идемпотентность партнера)|`rejected` или `completed` (если Duplicate)|
|`5xx / Timeout / Network Error`|Экспоненциальный retry (до 3 раз), затем отправка в DLQ|`retry_pending` → `failed`|
|`ACK не получен`|Ретрай по Kafka consumer offset, идемпотентность по `event-id`|Без изменений до подтверждения|

# **4. Топик и маршрутизация**

| Параметр        | Значение                             | Пояснение                                     |
| --------------- | ------------------------------------ | --------------------------------------------- |
| `Topic`         | `orders.returns.response`            | Иерархия: `<domain>.<aggregate>.<event>`      |
| `Partition Key` | `returnId` (string)                  | Гарантирует порядок обработки одного возврата |
| `Format`        | JSON (рекомендуется Schema Registry) | Валидация схемы при публикации                |
| `Retention`     | (оставлять пустым. заполняет разраб) | Хранение для реплея или отладки               |

# 📌**Структура сообщения**

**Key**
```json
"ret_8f3a1c9d"
```

**Headers**

| Header           | Тип             | Описание                                  |
| ---------------- | --------------- | ----------------------------------------- |
| `content-type`   | `string`        | `application/json`                        |
| `event-id`       | `string (UUID)` | Уникальный ID события для идемпотентности |
| `source-service` | `string`        | `partner-adapter-ms`                      |

**Value (Payload)**

| Поле            | Тип      | Обяз. | Описание                                        |
| --------------- | -------- | ----- | ----------------------------------------------- |
| `returnId`      | `string` | ✅     | ID возврата (совпадает с Key)                   |
| `partnerStatus` | `string` | ✅     | Статус обработки заявки: success, error, denied |
# 📌**Логика обработки**
```примечание
Адаптер партнера получает ответ от API партнера и  записывает его в topic Kafka.
```
1. Consumer забирает сообщение, проверяет `event-id` в БД (если уже был → `ACK` без обработки).
2. Выполняет маппинг полей.
3. Отправляет HTTP POST партнеру с таймаутом `5s` и `Idempotency-Key = returnId`.
4. При `2xx` → записывает сообщение в Kafka в обратный топик → коммитит offset.
5. При `409 (Duplicate)` → партнёр уже принял заявку. записывает сообщение в Kafka в обратный топик → коммитит offset.
6. При `5xx/Timeout` → инкрементирует `retry-count` в headers, републикует в топик `orders.returns.retry`, не коммитит offset.
7. При `retry-count > 3` → отправляет в DLQ, ставит статус `failed`, алертирует мониторинг.

Маппинг:

| Поле REST ответа партнера | Topic Kafka   | Описание                     |
| ------------------------- | ------------- | ---------------------------- |
| returnId                  | returnId      | ID заявки возврата           |
| status                    | partnerStatus | статус обработки от партнера |
